1.     iNTRODUCTION

1.1     Document Summary

This document provides a detailed functional specification of the API and user interface for version 3.0 of the User Authentication Gateway.

The User Authentication Gateway is a back-end user authentication system to which any HTTP-enabled web or stand-alone application may interface. The gateway will authenticate userid/password combinations and, in its default implementation, will create sessions for authenticated users that contain information about the user including (but not limited to) user name, staff membership, and beamline access permissions.

It will be incumbent upon individual applications to perform user authentication, either by forwarding the users browser to the gateways login page (in the case of web applications), or by submitting a userid/password combination directly (in the case of non-web applications). It will also be incumbent upon individual applications to check for session timeouts and to check other information about the user generated by the authentication method.

The gateway comes complete with a simple authentication method that may be used out-of-the-box with implementations of Blu-Ice and Web-Ice. The gateway is also extensible and you may add your own authentication methods.

1.2     User Authentication Gateway Clients

Clients of the software described in this document may be either of the following:

  1. Any web-based application that is accessible to users via web browser from any intranet or Internet-enabled location. Please note that the only direct interaction between the end-users web browser and the gateway itself will be the users login screen. All other interaction will be between the web-based applications server (whose IP address must be previously known to the gateway) and the gateway.
  2. Any non-web based application that would like to do user authentication and that is running on a system whose IP address is previously known to the gateway.

1.3     Platforms

1.       Server-side components of the User Authentication Gateway were implemented using Java Servlets and have been tested using Apaches Tomcat servlet container (versions 4.1 and 5.5). Please see section 2 for information on deploying the gateway on your Tomcat installation.

2.       User Authentication Gateway Clients (described above) may be implemented on any platform that can provide an HTTP connection to the gateway. The IP addresses of client systems (application servers or stand-alone applications) must be made known to the gateway via its configuration file.

3.       End-user browser-based components of gateway (currently limited to the login screen) will be implemented using simple HTML only.

4.       In order to access the User Authentication Gateway, end-users will be required to use current versions of web browsers that support secure (SSL) connections. In order to maintain the security of session ids and user information, it is strongly recommended that all applications use SSL to communicate between the application server and the end-users web browser. It is also recommended that end-users use web browsers be enabled for session-level cookies, although it will be left up to individual web applications if enabled cookies are a requirement. Web applications may choose to pass session ids to the gateway in the form of URL parameters instead of as cookies.

1.4     Cookies

Cookies are a common, and sometimes controversial, means by which a web-based application may store user, session, application state, or other data on an end-users system. Cookies are transmitted to by an application to a web-browser via a text-string in the HTTP header of a web page. Cookies may be stored permanently (or for a specified period of time) on a users system, or they may exist only for the life of the users current session (that is until the user closes the web browser). These temporary cookies are sometimes known as Session Cookies.

Session cookies are commonly used to maintain authentication data during an end-users session so that the end-user need only log in once, at the start of the session, to a web applications domain. This allows the end-user to go from page to page within an application, or from application to application within a domain, without having to endure multiple login screens. Many web application domains, such as Yahoo, telephone companies, banks, and credit card companies, require the end-user to enable session cookies on their browsers before the user may log in to an application. Other web application domains do not require cookies, choosing instead to store user authentication data within the URL of every web page.

Allowing cookies provides the end-user with easy access between web pages and applications, since session data will be automatically transmitted to all web applications within the defined domain. Cookies also provide a layer of security, in that session data is hidden from the user and sessions are limited to the users web-browser session. End-users need not worry about privacy concerns since the cookie disappears when the web browser is closed; no permanent data is written to the users system, and, in the case of the User Authentication Gateway, no 3rd-party cookies are used. Esthetically, the use of cookies provides a cleaner URL to the user, hiding much of the inner workings of session management.

URL-based session management also has certain advantages. Debugging a web-application is easier, since the developer can open a new browser and type the session id into the URL, thereby skipping a login step. It also allows the end-user to open multiple browsers on multiple computers, sharing a session across systems by manually keying (or cutting and pasting) the session id into the URL. In theory, the end-user could share the session id with a user-support person when requesting help. (For the security conscious, however, these are all arguments against URL-based session ids).

The User Authentication Gateway will allow client applications to work using either cookie-based or URL-based session data. Individual web applications may choose which method they will use.

2.     getting started

2.1     Tomcat Installation

The User Authentication Gateway has been developed and tested using Apache Tomcat versions 4.1 and 5.5. While the gateway should run with any servlet-specification compliant container, we recommend Tomcat if you are new to servlet containers.

1.       Download Tomcat from http://jakarta.apache.org/tomcat/index.html and follow its instructions for installation. Its default port for HTTP access is 8080, but can be changed in the Tomcat configuration file.

2.       You must enable SSL support in Tomcat for user access to the WEBLOGIN servlet. It is also highly recommended that web and stand-alone applications communicate with the gateway using SSL as well. See the SSL Configuration HOW-TO page in the Tomcat documentation for more information. The default port for HTTPS access is 8443, but can be changed in the Tomcat configuration file.

3.       Copy the file gatewayConfig.jar to Tomcats /conf directory use jar xvf gatewayConfig.jar to unpack its files into the directory.

2.2     Other Servlet Containers

If you are using a servlet container other than Tomcat for the User Authentication Gateway, please see its instructions for enabling SSL support and for deploying the gateway servlet context.

Copy the file gatewayConfig.jar into the containers configuration directory or some other location where only the container has access to them. You may need to change code references to catalina.base in AuthenticationInit.java, SMBAuth_SimpleUserDB.java, and SimpleAuth_Test1.java to the directory containing these files.

2.3     Gateway Configuration Files

  1. Modify the file AuthGatewayApps.xml to include valid application names which will be passed as parameters in gateway URLs. The file comes pre-configured with application names: SMBTest, BluIce, and WebIce.
  2. Modify the file AuthGatewaySystems.xml to include the hostname and IP address of any systems that will host web or stand-alone applications that utilize the gateway. The file comes pre-configured with hostname localhost at IP address 127.0.0.1.
  3. Modify the file AuthGatewayMethods.xml as follows if you are planning to use the gateways default functionality with Blu-Ice or Web-Ice: modify the domain tag to reflect the domain for which cookies will be valid, modify the login_ tags to point to HTML files that you would like to include in the HEAD and BODY of the WEBLOGIN page, and add any additional properties you would like the gateway to store in a user session. The default properties included are ones expected by Blu-Ice and Web-Ice.
  4. Modify the file SimpleUserDB.xml as follows if you are planning to use the gateways default functionality, especially with Blu-Ice or Web-Ice: modify the AllBeamlines tag to include the names of all your beamlines (ex. BL9-1;BL9-2), and add users defining all the properties shown in the sample users included as defaults.
  5. If you are planning to implement your own authentication method, please see section 5: Authentication Methods.

2.4     Gateway Deployment

For Tomcat installations, copy the file gateway.war to Tomcats /webapps directory. If Tomcat is configured for auto-deployment the gateway servlets will be loaded automatically. If not, use the Tomcat manager application to deploy and start the gateway application.

For non-Tomcat installations, please read the servlet containers instructions for deploying applications.

2.5     Sample Applications

A sample web application and a sample stand-alone application are included in the file gatewayTest.jar.

The web application can be run by extracting the file SimpleAuth_Test1.class from the jar file into a servlet context such as the examples context in Tomcat. SimpleAuth_Test1 is a servlet which will exercise the gateway via the WEBLOGIN, SessionStatus, and EndSession servlets.

The stand-alone application exercises the gateway using the AuthGateayBean utility class and the APPFORWARD servlet. It requires that you first extract the file authUtility.jar from gateway.war into whichever directory you place gatewayTest.jar. Run the application with the command (Unix):

java cp authUtility.jar:gatewayTest.jar AuthGatewayTest

or (Windows):

java cp authUtility.jar;gatewayTest.jar AuthGatewayTest

 

 

3.     Authenticating an end-user

An end-user of an application has been authenticated if the application receives a valid Session ID (SMBSessionID, in the case of SMB applications) as part of the end-users request. Please see section 4, Session Data, for information on retrieving the Session ID and determining its validity.

If the end-users request to the application does not contain a Session ID, or if the Session ID is not valid, then it is the applications responsibility to authenticate the user, as follows.

3.1     Web-Based Applications

Web-based applications may authenticate an end-user and create a new end-user session by redirecting the end-users browser to the gateway login page. At that page, the user will be prompted to enter a userid and password. If successfully authenticated, the browser will then be redirected back to the calling application.

If a request is made to your application without a Session ID cookie or URL parameter, or with an invalid Session ID, it is the application's responsibility to redirect the user's browser to the Gateway login page. The user's browser can be redirected by returning an HTTP response code of 302 (Object Moved) with the following location header added to the response header:

Location <AUTH_URL>

where <AUTH_URL> takes the following form:

https://<hostname:port>/gateway/servlet/WEBLOGIN?<P1>=<V1>&<P2>=<V2>&&<Pn>=<Vn>

where <hostname> is the name of the system on which the authentication gateway is running ("smb.slac.stanford.edu"), port is the SSL port number (if not the standard port 443), <P1>, <P2>,<Pn> are parameter names and <V1>, <V2>, and <Vn> are corresponding parameter values. All parameters, except for URL and AppName, are optional. When a parameter is not present in the AUTH_URL, its default value (as defined below) will be used.

The following lists the parameters that may be included in the URL, along with their default values.

URL=<url> - the fully-encoded URL to which the gateway will redirect the end-users browser upon successful authentication. If this URL needs to include parameters, they should either be passed as separate parameters (see below) or encoded into the URL using %3F for ?, %3D for =, and %26 for &, and %20 for space. There is no default value. If no URL parameter is included, an error message will be displayed on the users browser.

URLSession = <True or False> URLSession=True indicates that the gateway should include the Session ID as a parameter in the redirected URL, in addition to the Session ID cookie. Any other value indicates that the SMBSessionID should not be included as a parameter in the redirected URL. Default value: False.

Message = <string> - indicates that the message string should be displayed to the end-user as part of the login page. The message string must be encoded as with the URL parameter above. Default value: null

AppName = <string> - tells the gateway which web application is making the call. The AppName is checked against a list of valid application names that can call WEBLOGIN. If the parameter is not provided, or if its value is invalid, an error message will be displayed on the users browser.

SessionTimeout=<int> - sets the number of seconds that a session may remain inactive before automatically becoming invalidated. Default value: 1800. If the parameter is not provided or its value is not a valid integer, the default value will be used.

dbAuth = <True or False> - DEPRECATED. Default value: False. This parameter will be ignored in version 2.0 and above, in favor of the AuthMethod parameter, described below.

AuthMethod=<string> - tells the gateway which authentication method to use. See Section 5 for more information about authentication methods. <string> must match one of the authentication method names defined in the gateway's configuration data. If <string> does not match one of the authentication method names, an error message will be displayed on the user's browser. If this parameter is not included, the default authentication method will be used. Default value: null

Any additional parameters not defined here will simply be passed back to the calling application.

Examples of the redirection URL:

https://smb.slac.stanford.edu/gateway/servlet/WEBLOGIN?URL=https://smb.slac.stanford.edu/portal/servlet/VideoSystem&Message=Your%20session%20has%20expired.%20Please%20log%20in%20again.&AppName=VideoSystem

https://smb.slac.stanford.edu/gateway/servlet/WEBLOGIN?URL=https://myserver.slac.stanford.edu&URLSession=True&AppName=MyApp&MyParam=SomeParameterValue

https://smb.slac.stanford.edu/gateway/servlet/WEBLOGIN?URL=http://myserver.slac.stanford.edu/myapp/servlet/MyApp%3FMyParam%3DStuff%26MyOtherParam%3DMoreStuff&AppName=MyApp&AuthMethod=simple_user_database

Upon successful authentication, the first example will redirect the users browser to: https://smb.slac.stanford.edu/portal/servlet/VideoSystem where the Session ID (SMBSessionID) will be found in the header as a Cookie if the end-user's browser supports cookies.

The second example will include the Session ID in the URL and redirect the users browser to: https://myserver.slac.stanford.edu?MyParam=SomeParameterValue&SMBSessionID=<sessionid>

The third example will redirect the users browser to:

https://myserver.slac.stanford.edu/myapp/servlet/MyApp?MyParam=Stuff&MyOtherParam=MoreStuff and will attempt to authenticate using the simple_user_database authentication method.

3.2     Gateway Login Page

The Gateway Login Page will consist of the following elements:

4.       Header a header whose HTML content will be specified by the Authentication Method definition being used (see Section 5).

5.       HTML Body (top) HTML content specified by the Authentication Method definition.

6.       Application Message a message provided by the application for display to the user.

7.       Userid Field standard text box in which the user types in his or her user id.

8.       Password Field standard password text box in which the user types in his or her password. Only asterisks are displayed.

9.       Submit button submits the user information to the server for authentication. All information transmittal is encrypted via SSL.

10.   HTML Body (bottom) HTML content specified by the Authentication Method being used.

 

Successful authentication will cause a browser redirect to the URL specified by the URL parameter.

3.3     HTML Frames

If your web application is frame-based, you will find that redirecting a frame to WEBLOGIN will display the gateway login page within that frame. Redirect the frame instead to FRAMELOGIN, with all the same parameters as WEBLOGIN. FRAMELOGIN will return some JavaScript that will instruct the browser to redirect its parent frame to WEBLOGIN.

3.4     Non Web-Based Applications

For non-web applications that would like to create a session for a user, make an HTTP or HTTPS request to:

http(s)://<hostname:port>/gateway/servlet/APPLOGIN?userid=<userid>&passwd=<encodedid>& AppName=<application name> where <hostname> is the system on which the gateway is running, <port> is the port on which the gateway is listening for this request (if other than standard ports 80 or 443), <userid> is the userid in clear text, <encodedid> is the Base64 encoded hash of userid:password, and <application name> is the name of the calling application. All these parameters are required. Additional optional parameters are AuthMethod=<string> and SessionTimeout=<int> as described for WEBLOGIN. The default value for SessionTimeout for app logins is 259200 (72 hours). AppName must match a list of permitted applications, and the call to APPLOGIN must come from a trusted IP address as defined in the configuration files. If successful, the http response will include a Session ID cookie and the session information specified in section 4. If unsuccessful, the http response will consist of a response code 403 (Forbidden).

 

4.     Session Data

4.1     Session ID

Once an end-user has successfully logged in, a 128-bit session id will be created and returned. The name of the Session ID is found in the configuration data for the authentication method (see Section 5). The name of the Session ID for SMB applications such as Blu-Ice and Web-Ice will be SMBSessionID. For end-user browsers that allow session-cookies, the Session ID is stored in the browser, otherwise it may be returned as a URL parameter. Each request to your web application should include SMBSessionID as a cookie in the cookie header of the http request from the user's browser, or as a URL parameter.

Cookies are found in the HTTP header in the form:

cookie: <cookie1>=<value1>; <cookie2>=<value2>; ... <cookieN>=<valueN>

It will be your application's responsibility to extract the Session ID value from this cookie header.

If you are choosing to use URL-based session ids, then the redirection from the login page to your application will include Session ID as a URL parameter (in the form, for SMB applications, SMBSessionID=<string>. It will be the responsibility of your application to extract the session id from the URL and pass it to other pages within your application.

The Session ID may also be found in the text of the <BODY> of the HTML response to the APPLOGIN servlet.

If no Session ID value is found in the cookie header (or URL), then the user has not logged in, and should be redirected to the login page (as described above).

4.2     Session Lifespan

Sessions created for web-based applications (that is, those sessions created by calls to the WEBLOGIN servlet) will time-out after 30 minutes have passed without any activity on the session. Any activity on the session will keep it alive for another 30 minutes. The default value may be changed by use of the SessionTimeout parameter as described above.

Sessions create for non web-base applications (that is, those sessions create by calls to the APPLOGIN servlet) will time-out 72 hours after they are created. No activity is required on the session to keep it alive within that period. This default value may be changed by use of the SessionTimeout parameter as described above.

4.3     Session Sharing

A non web-based application may spawn a web-based application, sharing its session authentication data so that the user does not have to explicitly log in to the web-based application. It should be noted, however, that the web-based application will maintain its own session object, separate from that of the application from whence it came. This will allow the user to explicitly log out from the web application without affecting the parent application and vice-versa.

So, for example, if Blu-Ice were to create a session using the APPLOGIN servlet, it could then have a button that would bring up a web-browser with the Video System already logged in for the user.

An application can start a cloned web session by starting a browser and setting it to:

http(s)://<hostname:port>/gateway/servlet/APPFORWARD?URL=<app url>&AppName=<application name>&<sessionkey>=<session id>

where <hostname> is the system on which the gateway is running, <port> is the port on which the gateway is listening for this request (if other than the default ports of 80 or 443), URL is the URL of the web-based application, AppName is the name of the calling application, <sessionkey> is the name for session Ids defined in the authentication method (SMBSessionID) and <session id> is the calling applications session id. All parameters are required, and the AppName must be on the list of known applications, and the IP address of the calling application must be on the list of trusted systems as defined in the configuration files.

There is no planned mechanism by which a web-based application would spawn a non web-based application. That is, a user would not run Blu-Ice by clicking a link on a web page.

4.4     Session Validity and Data

Once you have found a Session ID string, you can check on the validity of the current session by making an HTTP or HTTPS request to:

http(s)://<hostname:port>/gateway/servlet/SessionStatus;jsessionid=<Session ID>?AppName=<application name>

or to

http://<hostname:port>/gateway/servlet/SessionStatus;jsessionid=<Session ID>?RecheckDatabase=True&AppName=<application name>

where <hostname> is the system on which the gateway is running, <port> is the port on which the gateway is listening for this request (if other than the default ports of 80 or 443), and <SessionID> if the session being queried.

The first form will check if the Session ID is currently valid. The second form will check if the session is currently valid, and will also re-query the appropriate database to refresh the additional user data stored in the user's session. This form should be used if theres a possibility of frequent changes to user data (such as accessible beamlines) during the course of a typical session.

Requests to this servlet will only be accepted from trusted computers listed in the AuthGatewaySystems.xml file found in the configuration directory. The AppName parameter is required, and it must be the name of a trusted application listed in the AuthGatewayApps.xml file.

Each request will receive a response consisting of a text-only web page with the following information found both as headers in the HTTP header response and as text within the response body:

Auth.SessionKey=<keyname> where <keyname> is the name of Session IDs as defined in the sessions authentication method. For the simple_user_database method, this value is SMBSessionID.

Auth.<keyname>=<SessionID> where <keyname> is the value returned in the Auth.SessionKey header and <SessionID> is the 128-bit session id. For example: Auth.SMBSessionID=0847ABDE13E1B9B55250C178C66A3B64

Auth.SessionValid=<status> where <status>=TRUE for a valid session, FALSE for an invalid session. For any response where the SessionValid value is not TRUE, the user must be redirected to the login page.

Auth.SessionCreation=<time> where <time> is the session's creation time expressed as milliseconds since midnight, Jan 1, 1970, UTC.

Auth.SessionAccessed=<time> where <time> is the last time the session was accessed, also expressed as ms since midnight, Jan 1, 1970 UTC.

Auth.UserID=<userid> where <userid> is the user id used to log onto the system

Auth.Method=<authentication method> where <authentication method> is the method used to create this session. For example: Auth.Method=simple_user_database.

Additional information will be returned as defined in the Authentication Method being used. For example, sessions created with the simple_user_database method will also return the following fields used by Blu-Ice and Web-Ice:

Auth.AllBeamlines=<string> where the <string> is a semi-colon delimited string of all beamliens available to Blu-Ice or Web-Ice.

Auth.RemoteAccess=<TRUE|FALSE>

Auth.Enabled=<TRUE|FALSE>

Auth.UserPriv=<priv>

Auth.UserName=<name>

Auth.Beamlines=<active beamlines> where <active beamlines> will be a string in the form: "bl;bl;bl" detailing which specific beamlines to which the user has access or ALL meaning that the user has access to all beamlines.

Auth.JobTitle=<string>

Auth.OfficePhone=<string>

Auth.UserStaff=<TRUE|FALSE>

4.5     Ending a Session

You can force a session to end by calling:

http(s)://<hostname:port>/gateway/servlet/EndSession;jsessionid=<SessionID>?AppName=<application name>

where <hostname> is the system on which the gateway is running, <port> is the port on which the gateway is listening for this request (if other than the default ports of 80 or 443), <application name> is the name of the calling application, and <SessionID> is the calling applications session id. All parameters are required, and the AppName must be on the list of known applications, and the IP address of the calling application must be on the list of trusted systems as defined in the configuration files.

Calling EndSession invalidates that session for all applications currently sharing it.

5.     Authentication Methods

5.1     Required Components

Two components are required to implement an authentication method in the User Authentication Gateway. The first component is an entry in the AuthGatewayMethods.xml configuration file that describes various aspects of the authentications method. The second component is a Java class that implements the AuthGatewayMethod interface. This section explains both components.

5.2     AuthGatewayMethods.xml

This file, stored in Tomcats /conf directory, describes the authentication methods to be used by the Authentication Gateway as well as the data that will be associated with a user's session. Multiple methods may be defined. The first method defined will be the default method used.

 

The following describes the entries for the simple_user_database method that comes with the gateway. You may use it as is, or modify it. You may also add entries for additional authentication methods. If you add a new method and would like it to be the default, be sure that it is the first method described in the file.

 

<?xml version="1.0" encoding="ISO-8859-1" ?>

<AuthGatewayMethods>

<AuthGatewayMethod>

<!-- the following elements are required -->

<!-- the name of the method -->

<name>simple_user_database</name>

<!-- the Java class that implements AuthGatewayMethod -->

<!-- interface and performs the authentication -->

<class>

edu.stanford.slac.ssrl.smb.SMBAuth_SimpleUserDB

</class>

<!-- the name of the session key -->

<keyname>SMBSessionID</keyname>

<!-- the domain in which session cookies are used -->

5.3     Authentication Method Class Files

For each method defined in the AuthGatewayMethods.xml file, create the class named in the <class> tag to implement the AuthGatewayMethod interface:

 

public interface AuthGatewayMethod {

 

public boolean authenticateUser( String userID,

String encodedUserPwd,

AuthGatewaySession auth );

 

public void addConfigurationData ( AuthGatewaySession auth );

 

public void updateAccessLog ( String appName,

String userName,

HttpSession session );

 

}

 

The authenticateUser function authenticates userID using the Base64 encoded encodedUserPwd and, if valid, populates the AuthGatewaySession object auth.

 

The addConfigurationData function checks for the additional configuration data specified in the authentication method and adds those properties to the AuthGatewaySession object auth.

 

The updateAccessLog function can be used to track usage of the web and stand-alone applications which use the gateway. Each call to WEBLOGIN, APPLOGIN, and SessionStatus triggers a call to this function. If you wish to do tracking, you could implement this function to write to a log file or database. If you do not wish to do tracking, you must still implement this function but have it do nothing (as shown in the simple_user_database method.

 

Please see SMBAuth_SimpleUserDB.java for an example implementation.

 

5.4     Package Files

The authentication classes must be packaged into a .jar file and placed in the appropriate Tomcat directory.

 

For example, the class file for the simple_user_database method is stored in smbAuthentication.jar and that file is copied to:

 

/<path to Tomcat>/webapps/gateway/WEB-INF/lib/smbAuthentication.jar

 

5.5     User Authentication Gateway Java Classes

 

The following are the Java classes that make up the gateway. See the API html document for more details.

 

5.5.1     Servlets

The following are found in gateway.war

  • APPFORWARD.class servlet used to clone a session for sharing between stand-alone and web applications.
  • APPLOGIN.class servlet used to authenticate a user and create a session from a stand-alone application.
  • EndSession.class servlet used to invalidate an existing user session.
  • FRAMELOGIN.class servlet which can be called from within an html frame to display WEBLOGIN in a browsers parent frame.
  • SessionStatus.java servlet used to query a session for validity and user information.
  • WEBLOGIN.java servlet used by web applications to obtain userid and password information from the end-user.

 

5.5.2     Utility Classes

The following are found in authUtility.jar

  • AuthBase64.class used to Base64 encode and decode userid:password strings
  • AuthGatewayBean.class used by applications which dont want to make direct calls to servlets for user authentication and session management.
  • AuthGatewayMethod.class interface for implemented authentication methods
  • AuthGatewaySession.class the gateway session object which stores data about the session and the user.
  • AuthMethodDescription.class an object each instance of which stores the description of an authentication method described in AuthGatewayMethods.xml
  • AuthenticationInit.class a gateway initialization class whose static method is called when each of the gateway servlets is first called.
  • SMBGatewaySession.class similar to AuthGatewayBean, this is a utility class used by SMB web and stand-alone applications.
  • SMBGatewaySession.class a Java Bean version of SMBGatewaySession

 

5.5.3     Authentication Method classes

The following is found in smbAuthentication.jar

  • SMBAuth_SimpleUserDB.class implements the simple_user_database authentication method.

 

5.5.4     Test Applications

The follwing are found in gatewayTest.jar

  • AuthGatewayTest.class a sample stand-alone application which uses AuthGatewayBean.class
  • SimpleAuth_Test1.class a sample web application which makes calls to the gateway servlets.
